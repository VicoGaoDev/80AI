# 后端安全整改清单

基于当前 `backend` 代码实现整理，目标是把“已有基础防护，但边界层偏薄弱”的状态，提升到可上线、可审计、可持续加固。

- 当前结论：基础防护已做
- 当前结论：边界安全不足
- 适合直接排期

## 最优先处理

先处理两类高风险问题：

1. `backend/app/services/cos_service.py` 的 SSRF 与路径穿越
2. 登录/注册缺少限流，容易被暴力尝试

## 概览

- 立即处理项：5
- 本轮加固项：6
- 上线前核查项：5
- 当前高风险缺口：2

## 当前防护现状

| 维度 | 状态 | 当前情况 | 风险提示 |
|---|---|---|---|
| 认证与鉴权 | 已具备 | JWT、`HTTPBearer`、角色校验、禁用账号校验 | info |
| 密码存储 | 已具备 | `bcrypt` 哈希 | success |
| 输入校验 | 部分具备 | 注册/登录/任务创建有基础校验 | warning |
| SQL 注入防护 | 相对较好 | 主要使用 SQLAlchemy ORM | success |
| 上传限制 | 部分具备 | 限制 MIME 和大小，但缺真实类型检测 | warning |
| 业务防刷 | 部分具备 | 任务提交锁和并发上限 | warning |
| HTTP 通用限流 | 缺失 | 未见接口级限流中间件 | danger |
| 安全头与主机校验 | 缺失 | 未见安全头配置或 TrustedHost | danger |

## 整改清单

### 立即处理项

- [ ] 修复图片拉取链路的 SSRF 风险，禁止访问内网、metadata、localhost 和私网网段。
- [ ] 修复本地图片读取的路径穿越风险，校验真实路径必须位于 `UPLOAD_DIR` 内。
- [ ] 给 `/api/auth/login` 和 `/api/auth/register` 增加限流与失败次数控制，优先按 IP 和账号双维度限制。
- [ ] 收紧 CORS，改为明确的前端域名白名单，不再使用 `allow_origins=["*"]`。
- [ ] 确保生产环境强制配置高强度 `SECRET_KEY`，禁止默认值启动。

### 本轮加固项

- [ ] 补充安全响应头，如 HSTS、`X-Content-Type-Options`、`Referrer-Policy`，并评估 CSP。
- [ ] 增加主机头校验，例如 `TrustedHostMiddleware`，防止 Host Header 滥用。
- [ ] 上传文件除校验 MIME 外，再做真实文件类型检测，避免伪造扩展名或 Content-Type。
- [ ] 为高成本接口增加通用限流，尤其是任务创建、图片上传、提示词反推。
- [ ] 优化 token 生命周期，评估缩短访问令牌时长并增加刷新机制。
- [ ] 确认生产环境禁用种子数据初始化，避免默认管理员账号被创建。

### 上线前核查项

- [ ] 记录登录失败、密码修改、管理员操作、配置变更等安全审计日志。
- [ ] 对登录暴力尝试、异常上传、任务洪泛和权限拒绝建立告警。
- [ ] 审查 `.env`、部署变量和第三方密钥权限，确保最小权限与定期轮换。
- [ ] 复核 `/uploads` 的暴露策略，确认敏感文件不会被匿名直接访问。
- [ ] 补几条有价值的安全回归测试，覆盖 SSRF 拦截、路径校验、登录限流和角色权限。

## 三阶段建议

| 阶段 | 建议周期 | 目标 | 完成标准 |
|---|---|---|---|
| P0 | 1-2 天 | 堵住高风险入口 | SSRF、路径穿越、登录限流完成并自测 |
| P1 | 本周内 | 补齐通用边界防护 | CORS、安全头、上传真实类型校验、接口限流完成 |
| P2 | 上线前 | 提升可审计与可运营性 | 安全日志、告警、配置审查、回归测试齐备 |
